### lm_head is NOT the embedding table
**Date:** 2026-04-12
**Context:** Computing output logits

`tie_word_embeddings: false` in `config.json`. Use `global/lm_head_weight.bin`
for logits, not the embedding table. They differ by up to 0.345.

### Instruct model needs the chat template
**Date:** 2026-04-12
**Context:** Prompt formatting for inference

Without the template, the model produces garbage. Format:
`BOS(128000) start_header(128006) user(882) end_header(128007) \n\n(271) {tokens} eot(128009) start_header(128006) assistant(78191) end_header(128007) \n\n(271)`

### M1 fixtures come from upstream; M2-3 fixtures are generated locally
**Date:** 2026-04-23
**Context:** Debugging test failures or planning fixture regeneration

`tests/data/test{2..7}_*.bin` are cherry-picked from the professor's upstream
repo -- there is no local script to regenerate them. Only
`tests/data/m2m3/*` is generated by `tools/gen_m2m3_fixtures.py`. If an M1
fixture seems wrong, re-pull from upstream, don't try to rebuild it.

### Don't drift back to prior hardware after committing to new
**Date:** 2026-05-02
**Context:** Multi-phase work where a verification step has selected target hardware

After Step 1 verified L4 as the target, I started Phase A on T4 because T4 had weights pre-loaded. Wrong: telemetry baselines must be on the same hardware as the phases they measure against. Once a verification step picks hardware, every subsequent phase runs there — even if the prior setup is cheaper or more convenient.

### KV cache without resident weights is correctness-only, not credit-eligible
**Date:** 2026-05-02
**Context:** Layer-streaming inference engines + autoregressive decoding

In a load_layer/unload_layer-per-layer engine, adding a KV cache produces correct outputs but zero speedup — each decode step still re-streams all N layers from disk to GPU. Mark such a feature in TODO.md as "logic shipped, perf gate pending Phase X" rather than closing the item.

### Question hardware before contorting code around its limits
**Date:** 2026-05-02
**Context:** Discovering a feature blocker tied to a specific GPU SKU

When a hardware capability is missing (e.g., T4 has no BF16 tensor cores), check whether a different SKU on the same cloud removes the blocker before designing software workarounds. A one-line GPU swap often beats N hours of dual-precision plumbing.

### Stop hedging when facts are deterministic
**Date:** 2026-05-02
**Context:** Communicating technical conclusions to the user

Phrases like "we'd need either A or B" imply uncertainty. If the technical facts force one answer, state it. Hedging is for product/scope decisions, not engineering conclusions that follow from documented hardware/software constraints.

### nvcc PATH varies by GCP deep-learning image
**Date:** 2026-05-02
**Context:** Building this project on a fresh GCP `deeplearning-platform-release` VM

The Makefile uses bare `nvcc` and relies on PATH. The T4 image (`common-cu128-...`) puts nvcc at `/usr/local/cuda/bin/nvcc`. The L4 image (`common-cu129-ubuntu-2204-nvidia-580`) puts it at `/usr/local/cuda-12.9/bin/nvcc` — no `cuda` symlink. Every SSH session must `export PATH=/usr/local/cuda-<ver>/bin:$PATH` before `make`. Confirm the path with `ls /usr/local/ | grep cuda` after first SSH.

### L4 has lower memory bandwidth than T4 — relevant only for non-resident paths
**Date:** 2026-05-02 | **Updated:** 2026-05-05
**Context:** Setting expectations when running this inference engine on different GPUs

L4 is 300 GB/s, T4 is 320 GB/s. With the FP32 streaming path, our matmul-heavy forward pass is memory-bandwidth-bound at B=1, so per-token decode is slower on L4 than T4. **However**, the resident BF16 path (Phase B, shipped 2026-05-02) eliminates the per-decode-step disk/PCIe streaming entirely, and L4's larger VRAM is what makes that path possible — so the practical decode latency on L4 is now far better than T4 ever was on this code. Phase D (BF16 WMMA) remains open and would close the remaining gap to peak throughput.

### Makefile arch override: use `make ARCH=sm_89`
**Date:** 2026-05-02
**Context:** Building with non-default `-arch` (e.g., sm_89 on L4)

The Makefile uses `ARCH ?= sm_75` and `NVCCFLAGS ?= -std=c++17 -O2` (both `?=`),
then appends `-arch=$(ARCH)`. To target L4, run `make ARCH=sm_89`. Do NOT pass
the full `NVCCFLAGS=...` string — the Makefile already builds it from `ARCH`,
and overriding `NVCCFLAGS` drops the `CUDA_ENABLED` define. (Earlier note about
`NVCCFLAGS :=` was correct for the previous Makefile and is now obsolete.)

### GCP `common-cu129` image lacks `build-essential`
**Date:** 2026-05-02
**Context:** First-time setup on a fresh GCP deep-learning VM

The L4 image (`common-cu129-ubuntu-2204-nvidia-580`) does NOT have `make` or `g++` preinstalled — `apt-get install -y build-essential` is required before any build. Setup scripts must run `apt-get update -qq` first or the install candidate is missing. The T4 image (`common-cu128-...`) had build-essential preinstalled, so this is image-specific.

### L4 spot capacity rotates by zone — iterate, don't pin
**Date:** 2026-05-02
**Context:** Provisioning g2-standard-4 + nvidia-l4 SPOT on GCP

Spot capacity for L4 was exhausted in us-central1-{a,b,c,f} simultaneously while us-east1-c had capacity. Provisioning scripts must iterate a list of zones (e.g., us-east1-c, us-east1-d, us-central1-a..f, us-west1-a, us-west4-a) until one accepts the create. Capacity rotates throughout the day; a zone that was full an hour ago may free up.

### Write v1 = correct, not v1-then-v2 "optimization"
**Date:** 2026-05-02
**Context:** Writing build/test scripts, runbooks, automation

When you have the knowledge to write the correct version, don't ship a "looks comprehensive" v1 and queue the real version for a follow-up pass. Apply parallelism, idempotency, and skip-when-unchanged on the first write. Cargo-culting an existing slow pattern is not the same as solving the problem. The user has to police every v1, and that erodes trust faster than missing features.

### Verify before quoting facts about this repo or this hardware
**Date:** 2026-05-02
**Context:** Producing concrete claims (file:line, build flags, GPU specs, clock numbers, behavior)

Failure pattern observed three times in one session:
1. Quoted `NVCCFLAGS :=` from a same-day learning entry — the Makefile had been changed to `?=` since the entry was written.
2. Quoted L4 boost clock as 2520 MHz — that is the RTX 4090; L4 is ~2040 MHz. Pulled from training-data memory of NVIDIA SKUs.
3. Framed "make weights resident" as a one-line tweak to fit a 10-step list — it is a `ModelWeights` rewrite (host pointers → persistent device buffers, plus removing the per-layer load/unload in `inference.cu`).

Rules:
- Before quoting any flag, file:line, function signature, or struct field, Read or grep the file in this turn. Do not rely on a learning entry, prior turn, or memory.
- Before quoting any hardware spec (clocks, bandwidth, FLOPS, VRAM), either run `nvidia-smi -q` on the actual machine or label it "from spec sheet, verify on box."
- If a requested format ("10 steps", "one line") would force misrepresenting scope, push back on the format. Do not silently compress a structural rewrite into a tweak.

Verification cost is cheap (one tool call). The cost of a wrong fact is the user re-reading every claim and losing trust in the rest of the answer.

### Do not use full regression as the performance-development loop
**Date:** 2026-05-03
**Context:** L4 validation for KV cache and bonus-performance work

Full M1+M2-3 regression is an acceptance gate, not the steering wheel for one
performance feature. For KV cache, resident weights, tensor cores, batching, or
attention work, create a targeted lane first: build incrementally, run one
purpose-built benchmark or parity check, print the exact timers, and fail fast
on forbidden old-path behavior. Run quick regression after the targeted lane
passes, then full regression once at the end.

### Custom-image fast boot is region-pinned; capture AFTER bin/llm exists
**Date:** 2026-05-05
**Context:** GCP custom images for repeated L4 provisioning

`tools/create_custom_image.sh` snapshots the boot disk into `<VM_NAME>-image`. Two non-obvious gotchas:

1. **Region pinning.** The image lives in the source disk's region. If `PREFERRED_ZONE` in `.l4-config.env` points to a *different* region on the next provision, GCP must copy the image cross-region first — silently — adding several minutes to "fast boot." The launching `gcloud compute instances create` blocks during the copy with no progress output unless redirection is removed. Fix: set `PREFERRED_ZONE` to a zone in the image's region.

2. **What's in the image.** `provision_l4.sh` only sets up data (weights, dump, fixtures, venv) — it does **not** push source or build `bin/llm`. Capturing the image directly after `provision_l4.sh` produces a "fast boot" that still doesn't have `bin/llm`, so `demo-start.sh` will fail. The correct capture point is *after* `tools/test_l4.sh --no-stop` rsyncs source and runs `make`. Workflow: provision → `test_l4.sh --quick --no-stop` → `create_custom_image.sh` → delete VM.

### L4 STANDARD vs SPOT — pick by failure mode, not just price
**Date:** 2026-05-05
**Context:** Provisioning g2-standard-4 + L4 on GCP for build-and-test work

SPOT is ~$0.20–0.30/hr but GCP can preempt the VM mid-build (observed: lost a 15-min provision run partway through the apt + weight-download phase). STANDARD is ~$0.72/hr (~3x) but won't be preempted. Rule of thumb: provision and build on STANDARD (where preemption ruins the work product), then either capture a custom image (so future boots are ~60s and preemption tolerance no longer matters), or flip back to SPOT for long REPL sessions. The script reads `PROVISIONING_MODEL` from `.l4-config.env`.

### Diagram prompt review must be adversarial and source-backed
**Date:** 2026-05-06
**Context:** TA key-area image prompts in `docs/diagrams/`

When reviewing diagram prompts for regeneration, do not stop at the current
image's visible mistakes. Re-read the prompt, verify every visible tensor shape
and operation against source, and look for prompt contradictions that can cause
the next image to drift. Especially check: title text appears in visible content,
mask inequalities are explicit about row/column axes, output-layer panels cannot
merge into decoder-block panels, and optional multi-token generation is described
as a loop of one-token decode steps unless the code supports `q_seq > 1` decode.
