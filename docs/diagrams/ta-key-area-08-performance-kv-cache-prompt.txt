Core story:
KV cache and resident BF16 weights solve two different performance problems.
KV cache reuses old K/V projections during multi-token decode. Resident BF16
weights avoid repeatedly moving layer weights from CPU/host storage to GPU.
In this repo, the practical fast multi-token path uses both. `forward_step`
returns a hidden vector; `lm_head + argmax` turns that hidden vector into the
next token.

Guardrail facts the image must obey, but should not render unless listed above:
- Part 2 does not require batching or KV caching.
- Assignment bonus asks about no-cache generation reprocessing
  `s0+1`, `s0+2`, ..., `s0+T`.
- Assignment bonus asks about attention score memory growing as `[s,s]`.
- `KVCache` stores K and V per layer as
  `[batch, max_len, NUM_KV_HEADS * HEAD_DIM]`.
- `S_MAX = 1024`, `NUM_LAYERS = 32`, `NUM_KV_HEADS = 8`,
  `HEAD_DIM = 128`, `kv_dim = 1024`, `EMBEDDING_DIM = 4096`.
- B=1 KV cache size is 256 MiB:
  `32 * 1024 * 1024 floats * 4 bytes * 2`.
- Prefill calls `forward_step` with `q_seq = prompt_len`.
- Decode calls `forward_step` with `q_seq = 1`.
- `forward_step` returns the final-RMSNormed hidden state for the last token.
- The token is produced after `lm_head` projection and greedy argmax.
- `kv_seq = cache.len() + q_seq`.
- K/V projection matmuls append new rows into the cache.
- `cache.advance(q_seq)` happens after all 32 layers finish.
- Attention still reads cached K/V over the valid prefix.
- The repo reuses one `AttentionScratch::d_S [q_seq,kv_seq]`; it does not
  allocate 32 score matrices at once.
- The one-token path may allocate `KVCache` for code reuse, but it has no
  decode loop and does not require KV caching for the milestone.
- `main.cpp` selects:
  - `max_tokens == 1` -> `generate_next_token`
  - `max_tokens > 1` -> `generate_tokens_resident`
  - multiple prompts -> resident batched decode with equal tokenized lengths
- `DeviceModelWeights` keeps per-layer BF16 weights resident in VRAM.
- Host `ModelWeights` still owns embeddings, final norm, and `lm_head`.
- Host/controller orchestration is not the same as CPU decoder compute; the
  decoder block work in `forward_step` is GPU/CUDA work.
- If a host icon is needed, use a generic "Host" rectangle, not a CPU chip.
- Performance claims should be tied to profiling/timers, not guessed.
