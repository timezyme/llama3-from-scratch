Title
Key Area 2: RMSNorm -> Q/K/V

Knowledge-transfer goal
Create a clean 16:9 CS265 TA-prep infographic that teaches the first attention pre-block:

`X -> RMSNorm -> X_norm -> Q, K, V projections`

The viewer should understand the main idea in 10 seconds and be able to explain it out loud in under 60 seconds.

Core story
RMSNorm prepares each token row before attention by rescaling the row's RMS magnitude and applying learned `gamma`; it does not change the shape and it does not mix tokens. The normalized matrix then goes through three learned projection matmuls to produce Query, Key, and Value roles. Q stays 4096-wide because there are 32 query heads, while K and V are 1024-wide because Llama 3 uses 8 shared KV heads for grouped-query attention. In this repo, K and V projection outputs are written directly into `KVCache`; Q is used for the current attention step.

Guardrail facts the image must obey but should not render unless listed
- Assignment base mental model is B=1: `X [s,d]`, `d=4096`.
- `s` is sequence length.
- `EMBEDDING_DIM = 4096`.
- `RMS_NORM_EPSILON = 1e-5`.
- `NUM_HEADS = 32`.
- `NUM_KV_HEADS = 8`.
- `HEAD_DIM = 128`.
- `KVCache::kv_dim() = NUM_KV_HEADS * HEAD_DIM = 1024`.
- RMSNorm formula: `rms(x)=sqrt((1/d)*sum_i(x_i^2)+epsilon)`.
- RMSNorm output formula: `y_i=(x_i/rms(x))*gamma_i`.
- `gamma [d]` is learned per element and must be applied.
- Epsilon is inside the square root.
- RMSNorm is row-wise and does not subtract the mean.
- RMSNorm maps `[s,4096]` to `[s,4096]`.
- Q projection math: `Q = X_norm * W_Q^T`, output `[s,4096] = [s,32*128]`.
- K projection math: `K = X_norm * W_K^T`, output `[s,1024] = [s,8*128]`.
- V projection math: `V = X_norm * W_V^T`, output `[s,1024] = [s,8*128]`.
- Checkpoint Q weight shape is `[4096,4096]`.
- Checkpoint K/V weight shape is `[1024,4096]`.
- Runtime transposed Q weight shape is `[4096,4096]`.
- Runtime transposed K/V weight shape is `[4096,1024]`.
- `src/model_weights.cpp` transposes checkpoint weights once at load time for the streamed FP32 path.
- `include/model_weights.h` documents transposed FP32 runtime shapes.
- `include/device_weights.h` documents resident BF16 transposed projection shapes.
- Resident BF16 path and streamed FP32 path use the same math.
- RMSNorm gammas remain FP32 in the resident path.
- `kernel/rmsnorm.cu` implements `gpu_rmsnorm(...)`.
- `gpu_rmsnorm(...)` uses one CUDA block per row, 256 threads, and shared-memory reduction.
- In the live code, `forward_step(...)` calls pre-attention `gpu_rmsnorm`, then Q/K/V projection matmuls.
- Streamed FP32 projections use `gpu_matmul_device`.
- Resident BF16 projections use `gpu_matmul_device_bf16_weight`.
- In the live code, Q is computed into `d_Q`.
- In the live code, K and V matmuls write directly to `cache.k_at(layer, len_before, b)` and `cache.v_at(layer, len_before, b)`.
- Panel 4 applies to both prompt prefill (`q_seq=s`) and one-token decode
  (`q_seq=1`). Do not label it as only a "current decoding step".
- Base assignment does not require batching or KV caching; this repo has optional batching and KV cache support.
- Live code uses `rows = batch * q_seq`; the poster should teach `[s,4096]` first and treat batching as hidden implementation detail.
- During prefill, K/V rows for the prompt are written.
- During decode, one new K row and one new V row are appended.
- RoPE happens after Q/K/V projection and rotates Q plus newly written K rows, not V. Do not make RoPE a main visual in this image.
- Q compares against K from the same layer's current/cached sequence, not against earlier layers.
- `lm_head` behavior is outside this infographic and should not be discussed.
